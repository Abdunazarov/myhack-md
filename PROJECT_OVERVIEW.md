# PRODUCT REQUIREMENTS DOCUMENT (PRD)
**Project Name:** Cradle LinkRouter (National Ecosystem OS)
**Document Purpose:** Бизнес-спецификация для последующей генерации Технического ТЗ (Technical Spec) и Системной Архитектуры.
**Core Philosophy:** Переход от ручной координации программ к системе самообучающихся программируемых сущностей (First-Class Reusable Entities), где данные прошлых когорт (Lost Intelligence) управляют автоматизацией в настоящем.

---

## 1. PRODUCT VISION & BUSINESS GOALS
* **Что мы строим:** Интеллектуальную операционную систему для национального акселератора (Cradle), автоматизирующую выдачу грантов, когортное менторство и передачу стартапов инвесторам.
* **Ключевая бизнес-инновация:** Отказ от статических баз данных в пользу "Живого Графа Отношений" (Dynamic Relationship Graph). Система использует петли обратной связи (Feedback Loops) из прошлых батчей для формирования рейтингов, мэтчинга и оценки рисков в реальном времени.
* **KPI платформы (для бизнеса):** 1. Увеличение пропускной способности Cradle в 10x (Zero-Touch Operations).
  2. Снижение времени на мэтчинг "Стартап-Ментор" с недель до секунд.
  3. Сокращение времени Due Diligence для VC-партнеров на 80%.

---

## 2. АКТОРЫ И РОЛИ (USER PERSONAS)
1. **Startup (Фаундер):** Подает заявки, взаимодействует с менторами, отчитывается по KPI.
2. **Mentor (Ментор/Эксперт):** Проводит трекинг, оставляет обратную связь, его рейтинг формируется скрыто на основе успехов его подопечных.
3. **Cradle Admin (Менеджер программ):** Контролирует "здоровье" экосистемы. Вмешивается только по алертам системы (управление по отклонениям).
4. **VC Partner (Внешний Инвестор):** Получает доступ только к валидированным паспортам выпускников Cradle (теплые лиды).

## 3. ФУНКЦИОНАЛЬНЫЕ МОДУЛИ И БИЗНЕС-ЛОГИКА

### МОДУЛЬ 1: Smart Intake & Auto-Routing (Умная воронка входа)
**Цель:** Автоматизировать пре-скрининг заявок на гранты и маршрутизацию проектов.

* **Feature 1.1: AI Pitch Audit (Пре-скрининг)**
  * **Input:** Заявка стартапа (Pitch Deck PDF, Financial Model CSV/XLSX, Sector, Stage).
  * **Business Logic:** Система сравнивает метрики заявки с историческими данными успешных выпускников Cradle в этой же нише.
  * **Output (Фаундеру):** Автоматический отчет с указанием отклонений (например, "Ваш CAC выше исторической нормы на 20%").
  * **Output (Системе):** Первичный скоринг проекта.
* **Feature 1.2: Ecosystem Routing (Маршрутизатор программ)**
  * **Trigger:** Заявка отклонена по жестким критериям гранта (Hard Constraints).
  * **Business Logic:** Вместо удаления данных, система ищет подходящую программу на уровень ниже (например, MYStartup Pre-Accelerator).
  * **Output:** Авто-перенос профиля стартапа в новую воронку без повторного ввода данных.

### МОДУЛЬ 2: Dynamic Cohort Orchestration (Управление когортой) *[CORE INNOVATION]*
**Цель:** Автоматизировать мэтчинг и трекинг на основе исторических данных.

* **Feature 2.1: Outcome-Based Mentor Entity (Динамический профиль ментора)**
  * **Business Logic:** Компетенции ментора не задаются текстом. Они вычисляются алгоритмически (Dynamic Tags) на основе результатов прошлых стартапов, которых он вел. 
  * **State Updates:** Если стартап 2024 года под менторством X успешно сделал B2B-пилот, ИИ начисляет ментору X скор `[Skill: B2B_Enterprise] = High`. Если стартап провалил маркетинг, `[Skill: Marketing] = Low`.
* **Feature 2.2: Autonomous Matching Engine**
  * **Input:** Текстовый запрос стартапа на решение проблемы (Roadblock) + Текущая стадия.
  * **Business Logic:** ИИ-оркестратор ищет не по ключевым словам, а по графу: выбирает ментора с наивысшим историческим KPI по решению *именно этой проблемы* для *именно этой стадии*.
  * **Output:** Сформированная сущность связи `Linkage_Entity` (Стартап <-> Ментор) со статусом `Active`.
* **Feature 2.3: Cohort Health Dashboard (Светофор здоровья)**
  * **Input:** Логи активности в системе (частота смены статусов задач, анализ тональности транскриптов звонков Google Meet - опционально).
  * **Business Logic:** Если `Linkage_Entity` не обновляется X дней, или сентимент-анализ выявляет конфликт.
  * **Trigger:** Смена статуса связи на `Requires_Intervention`.
  * **Output:** Push-уведомление администратору Cradle.

### МОДУЛЬ 3: Verified Handoff (Экосистемный выпуск)
**Цель:** Капитализировать успех Cradle и передать стартапы инвесторам (VC).

* **Feature 3.1: Cradle Verified Passport**
  * **Input:** Завершение программы (Trigger: Graduation).
  * **Business Logic:** Агрегация всего цифрового следа (оценки менторов, динамика KPI, результаты аудита) в неизменяемый цифровой документ (Immutable Record).
* **Feature 3.2: VC Targeted Push**
  * **Input:** Инвестиционный мандат VC-фонда (Сектор, Раунд, Локация).
  * **Business Logic:** Мэтчинг Verified Passport с мандатами VC-партнеров.
  * **Output:** Теплое интро в дашборде VC-инвестора ("Вам отправлен выпускник Cradle, валидация 100%").

## 4. БИЗНЕС-СУЩНОСТИ ДЛЯ ПРОЕКТИРОВАНИЯ БАЗЫ ДАННЫХ (DATA ENTITIES)

Для корректной технической архитектуры, система должна строиться вокруг следующих программных объектов:

1. **`Ecosystem_Project` (Сущность Стартапа):**
   - *Свойства:* State (Lead, In_Program, Graduated, Dead), Sector, Metrics_History (Time-series data).
   - *Поведение:* Накапливает "Ecosystem Passport" при переходе между программами.
2. **`Mentor_Node` (Сущность Ментора):**
   - *Свойства:* Availability_Capacity (для предотвращения выгорания), Dynamic_Skill_Matrix (массив тегов, обновляемый ИИ).
   - *Поведение:* Не имеет статического резюме. Обучается на результатах `Linkage_Entity`.
3. **`Linkage_Entity` (Сущность Отношений - *САМОЕ ВАЖНОЕ*):**
   - *Свойства:* Source (Startup ID), Target (Mentor ID), Goal, Health_Score, Final_Outcome (Success/Fail/Pivot), Feedback_Logs.
   - *Поведение:* Хранит историю взаимодействия. Именно данные из этой сущности используются для обучения `Mentor_Node` в будущих батчах.
